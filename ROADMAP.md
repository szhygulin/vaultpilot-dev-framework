# Roadmap

## Active

### #179 phase 3 — randomized curve study, 6 sizes × 3 replicates × 13 issues

**Why this scope** (vs. earlier "top 3 by size, K=5" draft): with only 3 distinct x values, a degree-2 polynomial has 3 coefficients — the curve passes exactly through the 3 size means regardless of K, so replicates only tighten standard errors at fixed points. They don't help the regression *choose* a curve shape. 6 sizes × 3 replicates fits a polynomial that actually has to choose, at the same total cell count.

**Methodology — random-sample trims**

Generated by `vp-dev research plan-trims` (committed in `83803a0`). Each (size, replicate) is a different random subset of the parent's sections fitting the byte budget. Across replicates every section appears in some small trims and is absent from others — the section-identity confound that broke phase 2's interpretation is averaged out.

**Sample**

6 sizes × 3 replicates = **18 dev-agents** (random-trim, generated fresh; not the operator-curated 9180–9189 set):

| Target size | Replicates |
|---:|---:|
| 6,000 B | 3 |
| 14,000 B | 3 |
| 22,000 B | 3 |
| 35,000 B | 3 |
| 50,000 B | 3 |
| 58,000 B | 3 |

× 17 issues across two real repos — 13 open + 4 synthesized controls:

| Repo | Open | Synthesized controls | Total |
|---|---:|---:|---:|

= **18 agents × 17 issues = 306 cells**.

(#179 is self-referential — keeping per the spec; flag if dispatched-agent behavior is anomalous.)

**Why synthesized controls (not closed-completed picks)**

The earlier draft picked closed-completed issues as ground-truth controls. Two problems:
1. Closed-issue bodies sometimes contain `Closes #N` / `See PR #M` self-references that pre-date the close — the body itself leaks the answer even with `--issue-body-only` blocking comment fetches.
2. Closed-completed picks bias toward the SUCCESS path; issues closed as wontfix / duplicate / not-planned were architectural pushback decisions and surface a different signal.

Synthesized controls fix both. The operator authors 2 fresh control issues per repo with predetermined "expected scope/disposition," files them as new GH issues, and the rubric becomes "agreement with the operator's predetermined disposition" — independent of any historical PR.

**Control issue authoring (operator)**

For each control, write the issue body + record a separate "expected disposition" the operator is willing to defend if the agent disagrees. Mix of decision shapes:

| Repo | Type | Expected disposition |
|---|---|---|
| vp-mcp | small concrete bug | implement: 1-line fix, exact file/line specified by operator |
| vp-mcp | substantive feature | implement: multi-file scope; expected file list recorded by operator |
| dev-agents | architectural ambiguity | pushback: scope is wrong-shape; agent should call out the mismatch |
| dev-agents | small UX fix | implement: bounded scope; expected file list recorded |

The "pushback" expected-disposition row is important — without it, all controls measure "did the agent ship?" and we miss the pushback-accuracy dimension. File the four issues with bodies that don't reference any PR, in any of `[Closes / Fixes / Resolves / See PR / merged via]` form. **Pre-screen each body**: grep the body text for `#\d+`, `PR `, `pull/`, `gh-actions` — reject if any match.

**Harness changes — IMPLEMENTED**

Five flags on `vp-dev spawn` (and forwarded by `vp-dev research curve-study`) added in commits `bcb70b7` and follow-up:

1. `--allow-closed-issue` — lifts the closed-issue gate (used here only if a control issue gets closed mid-run; fresh issues should stay open).
2. `--issue-body-only` — workflow Step 1 omits the comments fetch + suspends the "Issue Analysis" rule for the dispatch.
3. `--no-target-claude-md` — suppresses the live target-repo CLAUDE.md prepend so effective context size matches the per-agent CLAUDE.md size we're varying (CRITICAL — without this, "6 KB agent" is actually ~21 KB effective).
4. `--max-total-cost-usd` (curve-study only) — cumulative cost cap; aborts the run when reached.
5. `canUseTool` deny: when `--issue-body-only` is set, the gate refuses any Bash that fetches issue comments (`gh api .*/comments`, `gh issue view ... --comments`). Defense in depth on top of the workflow prompt.

Plus regex widening (`vp-dev/agent-[a-z0-9-]+/issue-N`) so plan-trims-generated agent IDs (`agent-916a-trim-50000-s50100`) don't break branch sweeping / open-PR matching.

Plus tooling:
- `vp-dev research register-trims --agents-spec X --trims-dir Y --tags-from agent-916a` registers the 18 generated dev-agents in the registry + copies their CLAUDE.md files into `agents/<id>/` automatically (replaces manual JSON editing).
- `vp-dev research plan-trims --clone-base /tmp/study-clones/{agentId}-{repo} --repos vaultpilot-mcp,vaultpilot-development-agents` emits the agents-spec with one entry per (agent × repo), so each leg has dedicated isolated clones.

**Total experiment after dispatch**

- Existing in `cells.json`: 76 cells (operator-curated trims from phase 1/2 + pilot — kept as historical record, NOT mixed with new data because the trim methodology differs).
- New (this phase): 306 cells.
- **Curve-fitting dataset: 306 cells** (the new randomized run feeds the regression on its own; legacy data is reference).
- **Combined archive: 382 cells**.

**Statistical power**

- 6 sizes × 3 replicates = 18 data points fitting a degree-2 polynomial → **df_residual = 15** (comfortable for F-test).
- Within-size variance averaging via K=3: SE reduced ~42% vs K=1.
- Compared to phase 2's degenerate 3-cells-per-size at 10 sizes: replicates per size are the difference that lets the F-test actually fire.

**Cost forecast**

- Real-repo issues are bigger / more complex than smoke-test. Phase-1 vp-mcp pilot ranged $0.30–$12.89 (median ~$1); phase-2 smoke-test mean was $5.12.
- Real-repo mean estimate: ~$7/cell.
- 306 × $7 = **~$2,140** point estimate; range **$1,500–$3,700** depending on cell-cost distribution.
- Cap each cell at $15 (`--max-cost-usd`) to bound runaway. Stop run if cumulative cost exceeds $3,500 + operator approval.

**Wall time**

- 8-way parallel (each agent has its own dedicated clone — contention-free), per-dev-agent serialization.
- 17 cells per agent serialized → ~150 min per agent at $7-mean ≈ ~9 min/cell.
- 18 agents / 8 slots = 2.25 waves → **~5.6 hours wall**.
- Two-phase split (one repo at a time) is acceptable if real-time pressure is high — same total wall, easier to monitor.

**Procedure**

1. **Author + file 4 control issues** in their respective repos (operator). Pre-screen each body for `#\d+` / `PR ` / `pull/` / `gh-actions` references; reject if any match.
2. **Generate trims**:
   ```bash
   vp-dev research plan-trims \
     --parent agent-916a \
     --sizes 6000,14000,22000,35000,50000,58000 \
     --replicates 3 \
     --output-dir feature-plans/issue-179-data/trims-phase3/ \
     --output-spec feature-plans/issue-179-data/agents-spec-phase3.json \
     --seed-base 2026 \
     --clone-base /tmp/study-clones/{agentId}-{repo} \
     --repos vaultpilot-mcp,vaultpilot-development-agents
   ```
3. **Register the 18 dev-agents** automatically:
   ```bash
   vp-dev research register-trims \
     --agents-spec feature-plans/issue-179-data/agents-spec-phase3.json \
     --trims-dir feature-plans/issue-179-data/trims-phase3/ \
     --tags-from agent-916a
   ```
4. **Set up 36 dedicated clones**: 18 of vp-mcp at `/tmp/study-clones/<agentId>-vaultpilot-mcp/` + 18 of dev-agents at `/tmp/study-clones/<agentId>-vaultpilot-development-agents/`. Total ~1.4 GB disk.
5. **Dispatch (vp-mcp leg)** — split agents-spec to entries with `repo === "vaultpilot-mcp"` first:
   ```bash
   vp-dev research curve-study \
     --agents-spec agents-spec-phase3-mcp.json \
     --target-repo szhygulin/vaultpilot-mcp \
     --issues 156,162,565,574,649,665,<control-1>,<control-2> \
     --parallelism 8 --dry-run \
     --no-target-claude-md --issue-body-only \
     --max-cost-usd 15 --max-total-cost-usd 1500
   ```
   → 144 cells (8 issues × 18 agents).
6. **Dispatch (dev-agents leg)**: same agents, different clone paths, different issues (`172,173,179,180,181,185,186,<control-3>,<control-4>`). → 162 cells (9 issues × 18 agents).
7. **Aggregate** into `feature-plans/issue-179-data/cells.json` with `phase: "phase3-randomized"`.
8. **Score + fit** both curves (`accuracyDegradationFactor` and `tokenCostFactor`) on the 306-cell dataset. Compute F-test p-values; if either curve has p < 0.05, hand-merge into `ACCURACY_DEGRADATION_SAMPLES` / `TOKEN_COST_SAMPLES`.
9. **Synthesized-control agreement scoring**: for the 4 control issues × 18 agents = 72 cells, compare each agent's proposed scope/disposition against the operator's predetermined disposition. Score 0/1 per cell. Two of four controls expect "implement" with a recorded file list; two expect "pushback" with a recorded reason.
10. **Cleanup**: deregister the 18 study agents, delete clones, delete trim files (or move to archive). Or `--keep` if a phase 4 follow-up is planned.

**Caveats / residual risk**

- **#179 self-reference**: dispatched agents may see this issue in the body of an issue they're given. Surface anomalous cells.
- **Repo issue diversity**: `vaultpilot-mcp` issues lean security/advisory; `vaultpilot-development-agents` issues lean tooling/feature-plan. Per-repo curves may differ; report both.
- **K=3 is the methodology floor**. Section-identity variance averaging is meaningful but not optimal; if results show high within-size dispersion, scale to K=5 in a phase 3.5 follow-up.
- **The 6 chosen sizes don't cover the full range** the placeholder curve was authored for (8KB–65KB per #177). Below 6KB and above 58KB the curve extrapolates.
- **Real-repo `--dry-run` still elides PR-create / push / comment side effects**. We're measuring "decide what to do," not "do it well end-to-end." A `--no-dry-run` follow-up at single-repo, single-size scale would close that gap, but at significant cost and side-effect risk.

**Future**

- **Phase 4** (if phase 3 produces a significant fit): harder issue set (curated cross-repo selection rather than "all open"), larger K (5–7), wider size range (4KB–80KB).
- **Phase 5** (if phase 3 quality signal is still flat): switch the y-variable to `tokenCostFactor` only (drop the quality dimension), accept that quality is unmeasurable at the model+specialty's current capability ceiling for this issue class.
